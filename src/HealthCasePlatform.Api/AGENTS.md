# AGENTS.md — Api layer

Caveman-strict. Layer rules only. Shared rules → root `AGENTS.md` + `ai/agents/Dotnet/ARCHITECTURE.md`.

## What lives here

Thin endpoints. Routing, DI injection, req→`IMediator.Send` call, result→HTTP/DTO map. DTO records. FluentValidation filters. Composition root (`Program.cs`).

## Dependency

→ Application + Infrastructure (composition root only). Endpoints inject **`IMediator` only** (+ nothing else) — never `AppDbContext`, never Application handlers directly. Host wires `AddMediator()` + `AddInfrastructure(config)` (no `AddApplication`).

## Hard rules

- Endpoint = route + name + filter + handler. Handler injects `IMediator` (+ nothing else) + `CancellationToken`. No business rules here. No EF. No `IOptions`. Handler builds a command/query record and `await mediator.Send(message, ct)`.
- HTTP mapping stays in handler → status codes / content types / problem-detail `detail` strings byte-for-byte preserved. Routes/names/filters/result types unchanged on refactor.
- Entity→DTO map at boundary (`ToCaseResponse`, inline `Select`). Never expose domain entity in response. DTOs = `sealed record` in `Cases/`.
- Transitions: handler returns `ErrorOr<RegulatoryCase>`. Map `Error.Type == ErrorType.NotFound` → `TypedResults.NotFound()` (404); else → `TypedResults.Problem(409, title:"Conflict", type:"…6.5.8", detail: error.Description)`. Ok → `TypedResults.Ok(ToCaseResponse)`.
- Input validation = FluentValidation endpoint filter (`ValidationFilter<T>`) → 400 first. Domain factory errors after that = unreachable via HTTP (superset). See root `AGENTS.md` Domain quirks.
- `AddValidatorsFromAssemblyContaining<Program>`. `AddProblemDetails()` + `AddExceptionHandler<GlobalExceptionHandler>()` (MUST register after `AddProblemDetails()` — handler injects `IProblemDetailsService`) + `UseExceptionHandler()` + `UseStatusCodePages`. `GlobalExceptionHandler` (maps malformed-body `BadHttpRequestException`/`JsonException` → 400 `application/problem+json` with no exception detail; unhandled → 500 with env-gated detail: `IHostEnvironment.IsDevelopment()` → `exception.ToString()`, else `null`; short-circuits `OperationCanceledException` without writing a response). Health checks `AddDbContextCheck<AppDbContext>` + `MapHealthChecks("/health")`. Swagger.
- **Auth pipeline:** `AddAuthentication()` (bare, **no default scheme** — no IdP yet) + `AddAuthorization()`; pipeline = `UseAuthentication()` + `UseAuthorization()` registered **AFTER** `UseExceptionHandler()`/`UseStatusCodePages()` and **BEFORE** endpoint mapping. Ordering load-bearing: it is why a 403/401 from the authz layer becomes `application/problem+json` (StatusCodePages fills the empty body) and why `GlobalExceptionHandler` (outermost) can catch auth-pipeline exceptions. `/health` + Swagger stay **open** — NOT under `RequireAuthorization`; do NOT add an authorization `FallbackPolicy` (it would close `/health`).
- **Role authorization:** every case endpoint has `.RequireAuthorization(...)`. Mutations = inline `RequireAuthorization(b => b.RequireRole(<role>))` (no named policies); reads (List/Get/History) = bare `.RequireAuthorization()` (any authenticated role, incl. Auditor). Roles in `Common/AppRoles.cs` (`static class AppRoles` — 5 `const string` + `All` array). Mapping: CaseOfficer → Create/Submit; ScientificReviewer → StartScientificReview; LegalReviewer → StartLegalReview; TeamLeader → RequestDecision/Approve/Reject + AssignScientific/AssignLegal; Auditor → read-only. AuthorizationMiddleware runs **before** endpoint filters → 403 precedes `ValidationFilter` (400). **Prod quirk:** bare `AddAuthentication()` has no default scheme → unauthenticated prod hit on a protected endpoint triggers `ChallengeAsync` w/ null scheme → `InvalidOperationException` → `GlobalExceptionHandler` → **500 `application/problem+json`, NOT 401**. Fix = wire a real/noop default scheme when a real IdP lands. Test host unaffected (default scheme `"Fake"`).
- **Review = two-stage authz (resource-based).** `StartScientificReview`/`StartLegalReview` gates are **widened**: `RequireRole(<ReviewerRole>, TeamLeader)` (coarse — 403s unrelated roles w/o a DB hit) **plus** a `ReviewAuthorizationFilter` (`Common/Authorization/`) that loads the case via `GetCaseQuery` and calls `IAuthorizationService.AuthorizeAsync(user, entity, <ReviewScientificRequirement|ReviewLegalRequirement>)`. `ReviewCaseAuthorizationHandler` (registered `AddSingleton<IAuthorizationHandler>` in `Program.cs`) `Succeed`s when `User.IsInRole(TeamLeader)` (override) **or** the matching `AssignedXReviewerId == userId` (resolved from `ClaimTypes.NameIdentifier`, fallback `Identity?.Name`). Unauthorized review ⇒ **403 `application/problem+json`** (filter returns `Results.StatusCode(403)` → StatusCodePages fills body — `TypedResults.Forbidden()` does NOT exist in .NET 10). 404-before-403 for unknown case: filter returns `TypedResults.NotFound()` (404 source **moved** from `MapTransition` to the filter; behavior identical). Review command/handler/domain signatures **unchanged**. One extra read per review action (filter `GetCaseQuery` AsNoTracking + command tracked `FindByIdAsync`).
- **Assignment endpoints** (TeamLeader-only): `POST /cases/{id:guid}/assignment/scientific` + `/assignment/legal`, each `.AddEndpointFilter<ValidationFilter<AssignReviewerRequest>>()` + `RequireRole(TeamLeader)`. `AssignReviewerRequestValidator` (`ReviewerId` `.NotEmpty().MaximumLength(100)`) is a **required superset** of the domain guard — the `.MaximumLength(100)` matches the `nvarchar(100)` column; without it a >100-char id passes filter+domain and EF throws `DbUpdateException` → 500. Handlers reuse `MapTransition` (NotFound → 404, Validation → 409, ok → 200).
- `MigrateAsync` startup block resolves `AppDbContext` (registered by `AddInfrastructure`) — unchanged. `public partial class Program;` at file end for `WebApplicationFactory<Program>`.
- `ApiFactory` test override: `RemoveAll<DbContextOptions<AppDbContext>>()` + re-add `AddDbContext`. Still applies — `AddInfrastructure` uses `AddDbContext`. No edit needed: `AddMediator` runs in the test host too, so integration tests exercise the real Mediator pipeline. **Auth test override:** `ConfigureTestServices` registers `AddAuthentication("Fake").AddScheme<AuthenticationSchemeOptions, FakeAuthHandler>("Fake", _ => {})` (test-only `FakeAuthHandler.cs`); `CreateClientWithRoles(params string[] roles)` sets `X-Roles` header; `CreateClientAs(string userId, params string[] roles)` sets **both** `X-User` and `X-Roles` (for resource-based authz tests where the caller's `NameIdentifier` must match `AssignedXReviewerId`). **All-roles default:** no `X-Roles` header ⇒ principal carries ALL roles ⇒ existing integration tests stay green w/ zero header edits. **`X-User` default:** no `X-User` header ⇒ `ClaimTypes.NameIdentifier`/`Name` = `"test-user"` (preserves every existing test). Both `X-User`/`X-Roles` defaults are **load-bearing** — altering either silently breaks the suite. (No tests-layer AGENTS.md — recorded here as pointer.)
